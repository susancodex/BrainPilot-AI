======================================================================
REAL STUDENT USER SIMULATION & FULL APPLICATION VALIDATION
==========================================================

You must not only review code.

You must actively use BrainPilot AI as if you are a real student using the platform daily.

Assume you are an actual university student who has never seen the application before.

Your responsibility is to validate the entire product from the perspective of a real end user.

Do not assume features work.

Verify them.

======================================================================
END-TO-END USER TESTING
=======================

Create realistic student scenarios and test the application completely.

Simulate the full student journey:

1. Landing Page
2. Registration
3. Email Verification
4. Login
5. Onboarding
6. Dashboard
7. Study Planner
8. AI Study Assistant
9. AI Tutor
10. Notes Management
11. Flashcards
12. Revision Planner
13. Productivity Tracker
14. Goals
15. Calendar
16. Analytics
17. Notifications
18. User Settings
19. Profile Management
20. Subscription Flow
21. Logout
22. Re-login

Every flow must be tested.

======================================================================
BEHAVE LIKE A REAL HUMAN USER
=============================

Do not test like a developer.

Test like a student.

Ask:

* Is this intuitive?
* Is this useful?
* Is this confusing?
* Is this too complicated?
* Is this feature actually helping students?
* Would a student understand this?
* Is the workflow logical?

Identify friction points.

Identify confusing UI.

Identify unnecessary steps.

Improve the experience.

======================================================================
FEATURE VALIDATION
==================

For every feature:

Verify:

* Works correctly
* Produces expected output
* Handles invalid input
* Handles edge cases
* Handles empty states
* Handles loading states
* Handles failures gracefully

If a feature does not work exactly as intended:

1. Identify root cause.
2. Explain issue.
3. Fix implementation.
4. Test again.
5. Verify solution.

Never leave partially working functionality.

======================================================================
FULL PROJECT VALIDATION
=======================

Review every part of the project.

Not only features.

Also verify:

* Architecture
* APIs
* Database
* Authentication
* Authorization
* Security
* Frontend
* Backend
* AI Systems
* Deployment
* Environment Configuration
* State Management
* Notifications
* Analytics
* Logging

Everything must be reviewed.

Everything must be validated.

Everything must be production-ready.

======================================================================
BUG DISCOVERY MINDSET
=====================

Assume bugs exist.

Attempt to find them.

Attempt to break the application.

Test:

* Invalid inputs
* Unexpected actions
* Rapid clicks
* Duplicate submissions
* Expired sessions
* Unauthorized access
* Network failures
* Missing data
* Corrupted data
* API failures

Think like a QA engineer.

Think like a malicious user.

Think like a student.

======================================================================
PRODUCTION READINESS REVIEW
===========================

Continuously ask:

If 100,000 students used this application tomorrow:

* Would it work?
* Would it scale?
* Would it remain secure?
* Would it remain maintainable?
* Would it provide a professional user experience?

If the answer is no, identify weaknesses and improve them.

======================================================================
NO ASSUMPTIONS RULE
===================

Never assume a feature works because code exists.

Never assume a feature works because a UI exists.

Never assume a feature works because an API exists.

Verify functionality through complete end-to-end analysis.

Only consider a feature complete after:

1. Code Review
2. Security Review
3. Functional Testing
4. Edge Case Testing
5. User Experience Review
6. Regression Testing

======================================================================
FINAL PRODUCT OWNERSHIP RULE
============================

Act as if you are the CTO, Lead Engineer, QA Lead, Security Lead, and Product Manager responsible for launching BrainPilot AI.

Your responsibility is not to generate code.

Your responsibility is to deliver a secure, scalable, maintainable, professional, production-ready educational SaaS platform that provides a high-quality experience for real students.

Continuously audit, review, improve, test, refactor, validate, and fix the entire system until no significant architectural, security, usability, performance, or functionality issues remain.
